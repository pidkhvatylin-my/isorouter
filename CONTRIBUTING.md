# Contributing

Thanks for your interest in improving isorouter! This is an npm workspaces
monorepo: `@isorouter/core` plus framework adapters for React, Vue and Svelte,
sharing test doubles via `@isorouter/test-utils`.

## Setup

```sh
git clone git@github.com:pidkhvatylin-my/isorouter.git
cd isorouter
npm install
npm run build
```

Node `^20.19.0 || >=22.12.0` is required (see `.nvmrc` for the version CI
uses).

## Working in a package

All commands below can be run from the repo root with `--workspace`, or from
inside `packages/<name>` directly.

```sh
npm run build -w @isorouter/core      # build (rolldown + tsc, or svelte-package)
npm run test -w @isorouter/core       # unit tests (vitest)
npm run test:e2e -w @isorouter/core   # Playwright e2e (requires `npx playwright install`)
npm run check -w @isorouter/core      # tsc --noEmit + publint + eslint + prettier --check
npm run lint:fix -w @isorouter/core
npm run format -w @isorouter/core
```

`@isorouter/core` additionally runs its e2e suite against both the native
Navigation API and the `@virtualstate/navigation` polyfill:

```sh
npm run test:e2e:native -w @isorouter/core
npm run test:e2e:polyfill -w @isorouter/core
```

Before opening a PR, make sure `npm run build` and `npm run check` pass for
every affected package, and that unit + e2e tests pass.

## Changesets

Releases and per-package `CHANGELOG.md` files are generated by
[Changesets](https://github.com/changesets/changesets). If your change
affects the published behavior of `@isorouter/core`, `@isorouter/react`,
`@isorouter/svelte` or `@isorouter/vue`, add a changeset describing it:

```sh
npx changeset
```

Follow the prompts to select the affected package(s) and bump type, then
write a short, user-facing summary — it becomes the changelog entry. Commit
the generated file in `.changeset/` along with your change.

Internal-only changes (docs, CI, tests, tooling, `@isorouter/test-utils`)
don't need a changeset.

## Pull requests

- Keep PRs focused — one logical change per PR.
- Add or update tests for any behavior change.
- Update the relevant package `README.md` if you change its public API.
- CI runs build, type/lint/format checks, unit tests and e2e tests on every
  PR — make sure it's green before requesting review.
